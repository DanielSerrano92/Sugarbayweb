1) Objetivo de la tarea
Implementar una solucion real para la seccion "Ayuda / Soporte" dentro de "Mi cuenta", de forma que el formulario deje de ser solo visual y pase a guardar solicitudes en base de datos asociadas al usuario autenticado, mostrando ademas el historial de solicitudes del propio usuario debajo del formulario.

2) Estado inicial de la seccion Ayuda / Soporte
- Existia formulario en UI con campos "Asunto" y "Mensaje".
- Al enviar, la API respondia con un flujo de mailto condicionado por SUPPORT_EMAIL.
- Si no habia configuracion, devolvia:
  "El canal de soporte por formulario no esta configurado en este entorno."
- No se guardaba ninguna solicitud en base de datos.
- No habia listado de solicitudes previas del usuario.

3) Problema detectado
- Se mostraba el mensaje de canal no configurado.
- El formulario no persistia datos en DB.
- No habia relacion de solicitudes con usuario en modelo de datos.
- El usuario no podia consultar historial desde la misma seccion.

4) Causa tecnica del problema
- Falta de modelo/tabla para soporte en Prisma.
- Endpoint `app/api/account/support/route.ts` implementado como fallback de `mailto`, sin `create` en DB.
- UI del dashboard sin estado/listado de solicitudes.
- La carga inicial de la pagina de cuenta no consultaba solicitudes de soporte.

10) Relacion creada entre usuario y solicitudes de soporte
- Relacion 1:N:
  - Un usuario puede tener muchas solicitudes (`User.supportRequests`).
  - Cada solicitud pertenece a un solo usuario (`SupportRequest.userId`).
- Consultas siempre filtradas por `session.userId`.

11) Como se guarda una solicitud
Flujo:
1. Cliente envia POST a `/api/account/support` con `subject` y `message`.
2. Servidor obtiene sesion via `getSessionUser()`.
3. Si no hay sesion -> 401.
4. Valida payload con `supportRequestSchema` (Zod).
5. Si valida -> crea registro con `createAccountSupportRequest(session.userId, parsed.data)`.
6. Responde exito con mensaje y solicitud creada.
7. Cliente muestra feedback de exito, limpia formulario y actualiza listado local al instante.

12) Como se obtiene el usuario autenticado
- En API route se usa `getSessionUser()` (`lib/auth/dal.ts`).
- El `userId` sale de la sesion en servidor.
- Nunca se acepta `userId` desde formulario/cliente.

13) Como se listan las solicitudes debajo del formulario
- SSR inicial en `app/account/page.tsx`:
  `listAccountSupportRequests(session.userId)`.
- Se pasan como `initialSupportRequests` al dashboard.
- En `renderSupportSection`, debajo del formulario se renderiza bloque:
  - Cargando solicitudes
  - Error al cargar
  - Estado vacio
  - Cards con asunto, mensaje, estado, fecha creacion y actualizacion
- Al crear nueva solicitud se inserta al principio del estado local sin recargar pagina.

14) Como se evita que un usuario vea solicitudes de otro
- GET y POST requieren sesion autenticada.
- Query de lectura:
  `where: { userId: session.userId }`
- Create de escritura:
  usa `session.userId` del servidor.
- No existe parametro de cliente para elegir owner de solicitud.

24) Incidencia posterior reportada y correccion aplicada
Error reportado:
- `TypeError: Cannot read properties of undefined (reading 'findMany')`
- Ubicacion:
  - `lib/repositories/account.ts` en `listAccountSupportRequests`
  - llamada desde `app/account/page.tsx`
- Contexto:
  - `prisma.supportRequest` aparecia como `undefined` en runtime.

Causa tecnica identificada:
- En entorno de desarrollo con HMR/singleton global, `lib/db.ts` podia reutilizar una instancia antigua de PrismaClient cacheada en `globalThis`.
- Esa instancia antigua podia no incluir el delegate nuevo `supportRequest` tras cambios de schema/client, provocando el error al invocar `findMany`.

Solucion aplicada:
- Se actualizo `lib/db.ts` para validar el singleton cacheado antes de reutilizarlo.
- Si el cliente global no expone `supportRequest`, se crea una nueva instancia de PrismaClient en lugar de reutilizar la antigua.
- Con esto se evita que el servidor siga trabajando con un cliente desactualizado respecto al schema generado.

Archivo modificado por esta correccion:
- `lib/db.ts`

Validacion de la correccion:
- Se ejecuto `npm run lint` sin errores de TypeScript/ESLint bloqueantes.
- El acceso a `app/account` deja de romper al cargar solicitudes de soporte cuando el cliente global previo estaba obsoleto.
