Proceso de implementacion de pagina Mi cuenta
Fecha: 2026-05-17

1. Objetivo de la tarea
Implementar una pagina de "Mi cuenta" completa, profesional y coherente con el estilo retro/80s/vaporwave/Windows 90 del proyecto, incluyendo:
- Perfil
- Mis pedidos
- Direcciones
- Seguridad
- Ayuda / Soporte

Ademas, mantener proteccion de acceso por autenticacion existente, validar en servidor y cliente, y no romper checkout/Stripe/tienda.

2. Estado inicial de la pagina Mi cuenta
- Existia `app/account/page.tsx` con una vista simple de perfil (solo lectura) y un bloque de accesos rapidos.
- No existian secciones completas de pedidos, direcciones, seguridad ni soporte.
- No existian rutas API especificas para gestion de cuenta.
- No habia formularios de cuenta con guardado/errores/estados detallados.

3. Problemas o carencias detectadas antes del cambio
- Sin edicion de perfil.
- Sin historial de pedidos en la zona de cuenta.
- Sin gestion CRUD de direcciones.
- Sin cambio de contrasena.
- Sin flujo de desactivacion/eliminacion de cuenta.
- Sin canal de soporte desde cuenta.
- Sin estados completos por seccion (carga/error/vacio/guardando/exito).

4. Arquitectura revisada
Autenticacion:
- `proxy.ts` protege `/account` y redirige a `/login`.
- `lib/auth/dal.ts` usa `requireSession` y `getSessionUser`.
- `lib/auth/session.ts` maneja cookie de sesion JWT.
- `lib/auth/actions.ts` maneja login/registro/logout y verifica `isActive`.

Usuario:
- Modelo `User` en `prisma/schema.prisma` con `firstName`, `lastName`, `email`, `username`, `phone`, `country`, `passwordHash`, `isActive`, etc.

Pedidos:
- Modelos `Order` y `OrderItem` existentes y usados por checkout.
- Repositorio de pedidos existente: `lib/repositories/orders.ts`.

Direcciones:
- Modelo `Address` existente con `type` (`SHIPPING`/`BILLING`), `isDefault`, etc.
- Checkout ya reutiliza direcciones en `lib/repositories/checkout.ts`.

Seguridad:
- Hash de contrasena con `bcryptjs`.
- Login invalida acceso cuando `isActive` es `false`.

Soporte:
- No existia backend real de soporte ni tabla dedicada.
- Se implemento validacion + fallback controlado por `SUPPORT_EMAIL`.

Checkout:
- Reutiliza direcciones existentes y snapshots en pedido.
- Se mantuvo compatibilidad del formato de direccion.

Prisma/base de datos:
- No se crearon tablas nuevas.
- Se reutilizaron `User`, `Order`, `OrderItem`, `Address`, `Cart`, `CartItem`.
- Sin migraciones nuevas.

API routes/server actions:
- Se revisaron patrones existentes (`/api/checkout`, `/api/search`, `/api/stripe/webhook`).
- Se crearon nuevas rutas API de cuenta.

12. Explicacion detallada de la seccion Perfil
Funcionalidad:
- Muestra nombre, apellidos, email, telefono, pais, usuario, fecha de nacimiento y fecha de registro.
- Permite editar campos permitidos: `firstName`, `lastName`, `username`, `email`, `phone`, `country`.
- Guarda via `PATCH /api/account/profile`.

Estados:
- Cargando perfil: contemplado en componente con `LoadStatus`.
- Perfil cargado: render normal.
- Error al cargar: panel de error con boton Reintentar.
- Guardando cambios: boton "Guardando...".
- Cambios guardados: feedback verde.
- Error al guardar: feedback rojo + errores por campo.

Seguridad:
- La actualizacion siempre se hace para `session.userId` en servidor.
- No se expone ni modifica otro usuario.
- Si `username` o `email` ya pertenecen a otro usuario, el guardado se bloquea con `409` y error de campo.

13. Explicacion detallada de la seccion Mis pedidos
Funcionalidad:
- Obtiene solo pedidos del usuario autenticado via `GET /api/account/orders`.
- Muestra:
  - numero de pedido
  - fecha
  - estado del pedido
  - estado de pago
  - total
  - productos y cantidades
  - metodo de pago
  - ciudad/pais de envio

Estado vacio:
- Si no hay pedidos: "Todavia no tienes pedidos."

Estados:
- Cargando pedidos (estado disponible en componente).
- Error al cargar pedidos (panel de error con reintento).
- Pedidos cargados correctamente.

14. Explicacion detallada de la seccion Direcciones
Funcionalidad:
- Lista de direcciones guardadas del usuario.
- Crear nueva direccion.
- Editar direccion existente.
- Eliminar direccion (con confirmacion `window.confirm`).
- Marcar direccion predeterminada.

Tipo de direccion:
- Soporta `SHIPPING`, `BILLING` y `BOTH` en formulario.
- `BOTH` crea/actualiza direccion para ambos tipos manteniendo compatibilidad con checkout.

Compatibilidad checkout:
- Reutiliza tabla `Address` existente.
- No se creo modelo paralelo.
- El checkout sigue leyendo direcciones desde el modelo actual.

Estados:
- Cargando direcciones (estado disponible).
- Sin direcciones: "No tienes direcciones guardadas."
- Error al cargar: panel de error.
- Guardando direccion: boton "Guardando...".
- Exito/error de alta/edicion/borrado con feedback.

15. Explicacion detallada de la seccion Seguridad
Cambiar contrasena:
- Campos:
  - contrasena actual
  - nueva contrasena
  - repetir nueva contrasena
- Validaciones:
  - obligatorias
  - coincidencia nueva/repetida
  - nueva distinta a la actual
  - minimo y complejidad (mayuscula, numero, especial)
- Se usa `bcrypt.compare` + `bcrypt.hash`.
- Nunca se devuelven contrasenas ni se loguean.

Eliminar/desactivar cuenta:
- Zona de peligro diferenciada visualmente.
- Confirmacion obligatoria escribiendo `ELIMINAR`.
- Accion ejecuta:
  - desactivar usuario (`isActive = false`)
  - limpiar direcciones y carrito
  - anonimizar email/username para evitar colisiones
  - mantener pedidos para integridad historica/legal
  - cerrar sesion
  - redirigir a Home

16. Explicacion detallada de la seccion Ayuda / Soporte
Funcionalidad:
- Texto explicativo de soporte.
- Formulario con `asunto` y `mensaje`.
- Validacion de no vacio y longitudes min/max.

Backend soporte:
- Si existe `SUPPORT_EMAIL`, API responde con `mailto:` preparado.
- Si no existe `SUPPORT_EMAIL`, se devuelve estado controlado indicando que el canal no esta configurado en ese entorno.
- La UI muestra feedback claro en ambos casos.

17. Explicacion del codigo implementado
Componentes principales:
- `app/account/page.tsx`: pagina protegida y entrega de datos iniciales.
- `components/account/account-dashboard.tsx`: UI, navegacion interna, formularios, estados y llamadas API.
- `lib/repositories/account.ts`: logica server-side de cuenta.
- `lib/validators/account.ts`: validaciones centralizadas.

Funciones principales:
- `getAccountProfile`, `updateAccountProfile`
- `listAccountOrders`
- `listAccountAddresses`, `createAccountAddress`, `updateAccountAddress`, `deleteAccountAddress`
- `changeAccountPassword`
- `deactivateAccount`
- `requestJson` en cliente para manejo uniforme de respuestas/errores.

Estados usados:
- Carga: `LoadStatus` por seccion (`loading`, `success`, `error`).
- Guardado/envio: flags como `profileSaving`, `addressSaving`, `passwordSaving`, `supportSaving`.
- Feedback: `success`, `error`, `warning`.
- Vacio: listas de pedidos/direcciones vacias con mensajes dedicados.

Validaciones:
- Zod en servidor para todos los payloads.
- Validaciones de campo y mensajes legibles.
- Validacion de unicidad de email y username en perfil contra otros usuarios.

Llamadas API:
- fetch a `/api/account/*` con JSON y manejo de `fieldErrors`.

Gestion de errores:
- Errores de validacion con `422`.
- Errores de autenticacion con `401`.
- Errores de recursos no encontrados con `404`.
- Mensajes legibles mostrados en UI.

19. Validaciones implementadas
Perfil:
- nombre/apellidos obligatorios
- username obligatorio con formato valido
- email obligatorio con formato valido
- pais obligatorio
- telefono valido si se informa
- bloqueo de guardado si username/email ya existen en otro usuario

Direcciones:
- nombre completo obligatorio
- calle obligatoria
- ciudad/provincia obligatorias
- codigo postal valido
- pais obligatorio
- telefono valido opcional
- tipo (`SHIPPING`, `BILLING`, `BOTH`)

Contrasena:
- actual obligatoria
- nueva obligatoria con complejidad
- confirmacion obligatoria y coincidente
- nueva distinta de actual

Eliminacion:
- confirmacion exacta con `ELIMINAR`

Soporte:
- asunto y mensaje obligatorios con minimos

20. Medidas de seguridad aplicadas
- Todas las rutas de cuenta validan sesion server-side.
- Todas las consultas/mutaciones filtran por `session.userId`.
- No se exponen hashes de contrasena ni secretos.
- No se confia en validacion cliente; Zod valida en servidor.
- Username y email son editables con validacion server-side de duplicados.
- Desactivacion de cuenta preserva relaciones de pedidos.
- Logout tras desactivacion.

21. Decisiones tecnicas tomadas y motivo
Decision 1:
- Reutilizar modelos existentes (`User`, `Address`, `Order`, `OrderItem`).
Motivo:
- Evitar duplicidad e incompatibilidades con checkout.

Decision 2:
- Implementar API routes especificas de cuenta.
Motivo:
- Separacion clara de responsabilidades y seguridad por endpoint.

Decision 3:
- Soft delete/desactivacion en lugar de borrado duro.
Motivo:
- Mantener integridad referencial e historico de pedidos.

Decision 4:
- Soporte via `SUPPORT_EMAIL` opcional.
Motivo:
- No inventar servicios externos ni credenciales; fallback documentado.

Decision 5:
- Layout de secciones con sidebar + selector movil.
Motivo:
- Mejor usabilidad desktop y responsive limpio en movil.

Actualizacion perfil (username + email editables) - 2026-05-17
- Se habilito la edicion de `username` y `email` en la seccion Perfil.
- Se amplio `updateProfileSchema` para incluir `username` y `email`.
- Se amplio `updateAccountProfile` para persistir `username` y `email`.

- La UI muestra errores por campo y no pierde los datos escritos cuando el guardado falla.
