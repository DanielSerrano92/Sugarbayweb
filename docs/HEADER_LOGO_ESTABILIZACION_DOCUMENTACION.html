Estabilizacion del header y logo de Sugarbay

Proyecto: Sugarbayweb
Fecha: 3 de mayo de 2026
Area: frontend, header, responsive, logo, navegacion

1. Error detectado

El header presentaba un comportamiento inconsistente segun el ancho real del viewport:

  1. En pantalla maximizada y zoom 100%, solo aparecia un logo.
  2. En ventana no maximizada pero estirada, aparecian dos logos.
  3. Los botones cambiaban de tamano entre ventana normal estirada y pantalla completa.
  4. Al entrar en responsive desaparecia la navegacion desktop, pero el logo debia seguir visible.

2. Causa raiz

2.1 Duplicacion del logo

El componente renderizaba dos enlaces a Home con espacio de logo: uno dentro de la navegacion desktop y otro absoluto para el modo compacto. Tras insertar la imagen en ambos slots, el DOM podia contener dos logos.

Ademas, en CSS existia una regla base para ocultar el logo compacto: .sb-header-compact-logo { display: none; }. Pero esa regla aparecia antes de .sb-header-logo-slot { display: grid; }. Como ambas clases tenian especificidad equivalente y .sb-header-logo-slot se declaraba despues, el display: grid podia volver a mostrar el logo compacto fuera de su breakpoint previsto.

2.2 Cambio de tamano de botones

El cambio de tamano venia de media queries especificas en torno a 1919px y 1920px. Por debajo de 1920px se aplicaban medidas menores; a partir de 1920px se aplicaban medidas mayores. Al maximizar una ventana que cruzaba ese limite, el header saltaba de un conjunto de tamanos a otro.

Tambien habia varias medidas dependientes de vw mediante clamp() en botones, gaps, padding, iconos y altura. Eso hacia que pequenos cambios de ancho modificaran la proporcion visual del header en desktop.

3. Solucion aplicada

  1. Se dejo un unico logo real renderizado en el header.
  2. Ese unico logo sigue siendo un Link href="/" a Home.
  3. Se sustituyo el logo del centro de la navegacion desktop por un espaciador invisible.
  4. Se eliminaron las clases de alternancia sb-header-compact-logo, sb-header-logo-slot-desktop y sb-header-logo-slot-compact.
  5. Se quitaron los breakpoints de max-width: 1919px y min-width: 1920px que provocaban el salto.
  6. Se fijaron medidas desktop estables para botones, iconos, gaps, padding y altura del header.
  7. Se mantuvieron reglas responsive solo donde el diseno cambia realmente: a partir de max-width: 1560px se oculta la navegacion desktop y el logo unico se reduce.

4. Codigo cambiado

4.1 Header antes de la correccion

El problema venia de tener dos lugares distintos donde podia aparecer el logo: 

<Link
  href="/"
  className="sb-header-logo-slot sb-header-logo-slot-desktop mx-1 shrink-0"
  aria-label="Ir a Home"
>
  <HeaderLogoImage />
</Link>

...

<Link
  href="/"
  className="sb-header-logo-slot sb-header-logo-slot-compact sb-header-compact-logo pointer-events-auto absolute left-1/2 top-1/2 z-10 -translate-x-1/2 -translate-y-1/2 shrink-0"
  aria-label="Ir a Home"
>
  <HeaderLogoImage />
</Link>

4.2 Header despues de la correccion

Ahora solo hay un logo real. El hueco central de desktop se conserva con un espaciador sin contenido interactivo. 

const HEADER_LOGO_SRC = "https://ik.imagekit.io/gq1enkszp/fotos/logo.png";

function HeaderLogoImage() {
  return (
    <Image
      src={HEADER_LOGO_SRC}
      alt="Sugarbay"
      fill
      priority
      sizes="(max-width: 639px) 70px, (max-width: 1560px) 106px, 156px"
      className="sb-header-logo-image"
    />
  );
}
<span className="sb-header-logo-spacer" aria-hidden="true" />

<Link
  href="/"
  className="sb-header-logo-slot sb-header-logo-link pointer-events-auto absolute left-1/2 top-1/2 z-10 -translate-x-1/2 -translate-y-1/2 shrink-0"
  aria-label="Ir a Home"
>
  <HeaderLogoImage />
</Link>

5. Archivos modificados

  • components/layout/header-client.tsx: se consolido el logo en un unico enlace a / y se anadio el espaciador desktop.
  • app/globals.css: se estabilizaron los tamanos desktop y se eliminaron las reglas que duplicaban/alternaban logos.

npm.cmd run lint

Resultado: sin errores de ESLint, TypeScript ni espacios invalidos en diff.

7. Como comprobarlo manualmente

  1. Abrir http://localhost:3000.
  2. Comprobar pantalla maximizada con zoom 100%: debe verse un solo logo.
  3. Salir de pantalla completa y estirar la ventana al maximo: debe seguir viendose un solo logo.
  4. Comparar los botones entre ambos estados: no deben saltar de tamano.
  5. Reducir la ventana hasta responsive: la navegacion desktop desaparece, pero el logo se mantiene visible.
  6. Hacer clic en el logo: debe navegar a /.

8. Fuentes consultadas

No fue necesario usar foros ni videos de YouTube. La solucion salio de inspeccionar el codigo local y contrastar el comportamiento con documentacion oficial.

  • Next.js Image Component. Se uso para validar fill, padre posicionado y sizes en imagenes responsive: https://nextjs.org/docs/app/api-reference/components/image
  • Next.js Link Component. Se uso para mantener la navegacion del logo con href="/": https://nextjs.org/docs/app/api-reference/components/link
  • MDN: Using media queries. Se uso para revisar como @media, max-width y min-width aplican estilos segun ancho del viewport: https://developer.mozilla.org/docs/Web/CSS/CSS_media_queries/Using_media_queries
  • MDN: width media feature. Se uso como referencia para entender consultas de ancho y sus variantes min-width / max-width: https://developer.mozilla.org/en-US/docs/Web/CSS/%40media/width
  • MDN: object-fit. Se uso para justificar object-fit: contain y evitar deformacion del logo: https://developer.mozilla.org/en-US/docs/Web/CSS/object-fit
  • MDN: transform. Se uso para revisar el impacto de transform en elementos posicionados y aplicar el hover en la imagen interna sin romper el centrado del enlace: https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/Properties/transform
